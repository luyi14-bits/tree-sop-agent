2.1 Loop Engineering — 2026 年核心话题

核心公式：Prompt 决定 Agent 如何开始，Context 决定 Agent 能看到什么，Loop 决定 Agent 最终能走多远。

一个完整的 Agent 循环：Think（思考）→ Act（执行）→ Observe（观察）→ Verify（验证）→ Evolve（演进）→ Repeat（重复）

业余 demo 和生产级系统之间最大的差距，就在这些"看起来无聊但决定生死"的工程细节上。Loop Engineering 的五个核心问题：

问题对应工程关键挑战
Loop 如何运行任务拆解 + 工具调用复杂任务的递归分解策略
Loop 如何停止退出条件 + 完成机制避免无限循环，检测收敛
Loop 如何验证中间结果检查自动质量评估指标
Loop 如何恢复失败恢复 + 断点续跑检查点粒度与恢复成本
Loop 如何调试日志 + 状态记录全链路可回放

2.2 Agent / Framework / Harness 三层本质区别

层级本质类比代表
Agent智能执行器 — 接收任务、调用工具、产出结果工厂里的工人ChatGPT, Claude Agent
Framework组件库 — 提供构建 Agent 的积木和编排能力工厂的流水线设计LangGraph, CrewAI
Harness操作系统 — 为 Agent 提供稳定的运行环境工厂的电力+安防+质检系统OpenAI Harness, Tree-SOP Harness
关键洞察：大多数开源框架只做到了 Framework 层，缺乏 Harness 层的硬约束和稳定性保障。这正是 AgentHarness 的差异化机会 — Harness 层（ToolGuard + LOOP SOP 5 级门禁）是真正的技术壁垒。

三、七大架构模式深度解析

2026 年主流 Agent 框架可归纳为 7 种核心技术路线。每种路线有其设计哲学、适用场景和固有局限。

📊
模式一：图状态机（Graph State Machine）
代表：LangGraph · Spring AI Alibaba GraphRuntime

设计哲学

将 Agent 工作流建模为有向图：节点（Node）= 计算单元，边（Edge）= 状态流转路径，共享状态（State）在节点间传递和累积。每个状态转换都是显式的、可审计的，没有任何隐式魔法。

核心技术要素

要素说明
StateGraph通过 TypedDict/Pydantic 定义 State Schema，每个字段配置 Reducer 控制合并策略
Pregel 执行引擎受 Google Pregel 论文启发的超步（Superstep）执行模型，同一超步的节点自动并发
Conditional Edges通过函数动态决定下一步路由，支持复杂分支逻辑
Send 原语支持动态 fan-out 到多个节点（Map-Reduce 模式）
Durable Execution每步执行后自动持久化 Checkpoint，崩溃可从断点恢复

代码范式

from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]  # 追加模式合并
    next_action: str
    iteration_count: int

workflow = StateGraph(AgentState)
workflow.add_node("planner", planner_node)
workflow.add_node("executor", executor_node)
workflow.add_conditional_edges("reviewer", route_decision, {
    "continue": "planner",
    "done": END,
    "retry": "executor"
})
app = workflow.compile(checkpointer=MemorySaver())

优劣分析

优势劣势
✅ 精确控制每一步执行路径❌ 认知负担重（需时刻思考状态在哪）
✅ 完整审计轨迹，满足合规需求❌ Hello World 级示例需 ~60 行代码
✅ Durable Execution 保障生产可靠性❌ 图思维对线性脚本工程师门槛高
✅ LangSmith 全链路可视化追踪❌ 过度显式导致简单场景也需大量样板

适用场景 金融合规审批流 · 多步数据管道 · 需要人工介入的生产系统 · 任何要求精确控制和完整审计轨迹的场景

生产案例 Uber · LinkedIn · Klarna — 均在 LangGraph 上稳定运行超一年

👥
模式二：角色团队（Role-Based Teams）
代表：CrewAI · MetaGPT

设计哲学

核心洞察：大多数现实工作流本质上是"一群人分工合作完成一个目标"。直接用这种自然语言描述 Agent 系统 — Agent（角色）+ Task（任务）+ Crew（团队）+ Process（流程）。

三种执行模式

模式机制生产可靠性
Sequential任务按声明顺序执行，前一个输出自动注入后一个上下文✅ 稳定可靠
HierarchicalManager Agent 自动决策子任务分配给哪个 Worker⚠️ 已知缺陷：Manager 经常分错任务
Concurrent多 Agent 并发执行，结果聚合✅ 适合独立子任务

⚠️ 生产实践警告：2026 年生产实践暴露了 Hierarchical 模式的严重缺陷。GitHub Issue #4783 真实报告显示：Manager Agent 经常将任务分配给错误的 Worker，甚至完全忽略特定 Agent 的专长。核心问题是 Manager 的协调逻辑依赖 LLM 语义理解，在复杂场景下不够可靠。

代码范式

from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role='高级研究员',
    goal='深度挖掘 AI Agent 框架的技术细节',
    backstory='你是一位有10年经验的技术研究员',
    tools=[search_tool, read_tool],
    allow_delegation=True
)

task = Task(
    description="研究 LangGraph 与 CrewAI 的核心架构差异",
    agent=researcher,
    expected_output="一份包含架构图解释的详细技术对比报告"
)

crew = Crew(
    agents=[researcher, writer],
    tasks=[task],
    process=Process.sequential
)

适用场景 快速验证 Agent 概念 · 内容生产流水线 · 客户服务分级路由 · 周期不超过 5 分钟的中等复杂度工作流

迁移趋势 业界最常见的迁移路径：CrewAI 原型验证 → 遇到控制流天花板 → 迁移到 LangGraph 生产加固

模式三：SuperAgent 运行时（SuperAgent Runtime）
代表：DeerFlow 2.0 (ByteDance) · OpenAI Swarm

设计哲学

如果说 LangGraph 给你积木，CrewAI 给你说明书，DeerFlow 给你一栋已经建好的、配备家具和家电的房子 — 底层仍然是 LangGraph 的"钢筋混凝土"，但你可以直接入住。

核心架构：Supervisor + 专化 SubAgent

用户目标
    ↓
[Supervisor Agent] ← 接收目标，制定计划，分配任务，汇总结果
    ├── [Researcher Agent] ← 网络搜索、文档分析、信息检索
    ├── [Coder Agent] ← Python 代码生成、沙箱执行、调试
    └── [Reporter Agent] ← 结果综合、报告生成、格式输出

双层内存架构

内存层内容作用
工作记忆当前 Agent 调用的活跃上下文严格控制在 Token 限制内
归档记忆完整任务历史存储超出部分序列化到文件系统，语义检索取回

独特优势 原生支持跨越数小时、涉及数十个子任务的长时程工作流，Docker 沙箱隔离保障安全

适用场景 企业深度研究报告生成 · 自动化代码审查与重构 · 竞品分析与市场调研

模式四：执行器模式（Executor Pattern）
代表：Microsoft Agent Framework

设计哲学

"契约优先、可观测优先、稳定性优先"。不重新发明轮子，而是把 Agent 核心语义（消息、线程、运行、工具、工作流、上下文）提炼为稳定的架构脊柱，外围保持演化能力。

分层架构

dotnet/src/
├── Microsoft.Agents.AI.Abstractions/  # 核心抽象层
│   ├── AIAgent.cs                     # Agent 基类
│   ├── AgentThread.cs                 # 线程抽象
│   └── AgentRunResponse.cs            # 响应模型
├── Microsoft.Agents.AI/               # 核心实现

│   └── Middleware/                    # 中间件系统
├── Microsoft.Agents.AI.Workflows/     # 工作流引擎
│   ├── Executor.cs                    # 执行器抽象
│   ├── Workflow.cs                    # 工作流定义
│   └── Checkpointing/                 # 检查点系统
└── Microsoft.Agents.AI.Workflows.Declarative/  # 声明式 YAML

核心特征

特征说明
双语言战略.NET 和 Python 两套完整实现，API 语义一致
执行器模式工作流基于 Executor 构建，RouteBuilder 声明式消息路由
中间件架构借鉴 ASP.NET Core，支持拦截、过滤、审批
OpenTelemetry 原生每个执行器自动生成 Trace 和 Metrics
声明式 YAML支持 YAML 定义工作流，低代码友好

适用场景 企业级开发 · .NET 生态 · 需要合规审计和分布式追踪的大型组织

模式五：类型安全 Agent（Type-Safe Agent）
代表：PydanticAI

设计哲学

来自 Pydantic 团队。核心主张：类型安全是生产级 Agent 的基础。基于 Pydantic 的运行时验证和类型推导，确保 Agent 输入输出、工具参数、状态结构在编译期和运行期双重保障。

核心特征

特征说明
模型无关支持 OpenAI、Anthropic、DeepSeek、Ollama 等，统一接口
Pydantic 验证所有输入输出自动校验与转换，类型错误在运行时立即暴露
依赖注入原生支持 DI，工具函数可注入类型安全的上下文
结构化输出直接返回 Pydantic Model，无需手动解析 JSON
轻量级核心理念是"做减法"，只提供生产级 Agent 真正需要的东西

工程质量 被 Loop Engineering 研究列为推荐学习的开源项目之一，代码质量极高

✨
模式六：极简代码优先（Minimal Code-First）
代表：smolagents (HuggingFace) · OpenAI Agents SDK

设计哲学

核心理念：Agent 不需要复杂的框架。smolagents 核心代码不到 1000 行，提供最小抽象。Agent 的"逻辑"直接用 Python 代码表达，而非 XML/YAML 配置。

核心特征

特征说明
Code-as-ActionAgent 的行动直接是 Python 代码，而非 JSON 工具调用
最小抽象没有复杂的类层次、没有图、没有角色定义
教育友好代码量小，适合学习和理解 Agent 原理
快速原型几行代码即可运行一个 Agent

局限 缺乏生产级的状态管理、检查点、可观测性能力，适合原型验证而非生产部署

五、12-Factor Agents 与 Context Engineering

5.1 12-Factor Agents（HumanLayer）

由 HumanLayer 创始人 Dex Horthy 在 2025 年 4 月提出，致敬 Heroku《12-Factor App》。这是构建可靠 LLM 应用的工程原则集，截至 2026 年 6 月已成为行业事实标准。

核心定义：Agent = prompt × context window × control flow × structured outputs × event log × HITL

#原则核心要求
1Natural Language Context上下文是自然语言，不是结构化配置
2Agent LoopAgent 本质是一个循环：接收→思考→行动→观察→重复
3Tools as Functions工具是类型安全的函数，不是魔法字符串
4Structured Outputs输出必须是结构化的，可验证的
5Environment ManagementAgent 运行环境必须可管理、可隔离
6Context Engineering主动管理上下文，而非被动塞入
7Compaction长对话必须压缩，保留关键信息
8State Management状态必须可持久化、可恢复
9Human-in-the-Loop关键决策必须有人工介入点
10Tool Guardrails工具调用必须有权限控制和约束
11Event Log所有操作必须记录事件日志
12Graceful Degradation失败时优雅降级，而非崩溃

5.2 Context Engineering 五大模式

2026 年 Agent 工程的核心挑战：如何在有限的注意力预算（Token Window）内，管理好 Agent 能看到的上下文。

模式机制代表实现适用场景
Progressive Disclosure按需加载上下文，不一次性塞入所有信息Agent Skills（Anthropic）Skill 库庞大时
Context Compression滑动窗口 + 摘要压缩，保留关键信息Tree-SOP ConversationCompressor长对话场景
Context Routing根据意图路由到不同的上下文子集Tree-SOP IntentRouter (HyDE)多领域 Agent
Evolved RetrievalAgentic RAG / Graph RAG / Self-RAGLlamaIndex知识密集型任务
Tool ManagementMCP 协议统一工具管理MCP（跨框架标准）工具生态扩展
AgentHarness 对标情况：已在 Context Compression（三模式压缩）、Context Routing（HyDE 路由）、Tool Management（MCPClient + ToolGuard）三个模式上实现了对标。Progressive Disclosure 通过 Skill 按需加载部分实现。

6.2 七大模式中的 Tree-SOP 定位

AgentHarness 的架构归属：不属于任何单一模式，而是Harness + 混合编排的创新路线。

  • Harness 层：对标 Microsoft Agent FW 的执行器模式，但更强（ToolGuard 三层拦截 + LOOP SOP 5 级门禁）
  • 编排层：融合 Sequential / Parallel / Hierarchical 三种模式
  • 记忆层：四层记忆体系，超越所有竞品
  • Skill→Agent 映射：Superpowers 理念的独立框架实现

📊 战略结论

AgentHarness 在 Harness 硬约束、四层记忆、缓存优化、Skill→Agent 映射 四个维度拥有真正的技术壁垒，这些是竞品不具备或不重视的。框架化的核心任务不是追赶 LangGraph 的图编排精度或 CrewAI 的社区规模，而是把已有的差异化能力封装成可复用的框架 API，让其他开发者能基于 Tree-SOP 的 Harness + 记忆体系构建自己的 Agent。

一句话定位：Tree-SOP 不是又一个 LangGraph 或 CrewAI，而是唯一一个把 Harness 硬约束和 DeepSeek 缓存优化做到框架级别的 Agent 框架。
